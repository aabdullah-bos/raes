# RAES Execute

## Durable Decisions

| Decision | Rationale | Date |
|----------|-----------|------|
| CLI tool enforces one slice per invocation; no multi-slice batch execution | PRD explicitly states "Does not support automated multi-slice/multi-milestone batch execution (one slice per invocation)." This boundary prevents drift and ensures disciplined, step-by-step advancement with human oversight at each decision point. | 2026-05-03 |
| Artifact files must never be merged, overwritten, or reinterpreted outside strict RAES rules | PRD states "Does not merge, overwrite, or reinterpret artifact files outside the strict rules defined in RAES." This is non-negotiable for maintaining artifact integrity and trust in system constraints, decisions, and product intent. | 2026-05-03 |
| CLI halts execution if artifact boundary is violated or unknown artifact is referenced in config | PRD Safety/Drift Guards (Priority: High) requires "Disallow execution if artifact boundary is violated or unknown artifact is referenced in config." Prevents silent corruption and forces human resolution of ambiguity before proceeding. | 2026-05-03 |
| Durable constraints promoted from decisions.md must be referenced in system.md (and vice versa); tool must validate or warn when boundary is unclear | PRD Boundary/Artifact Discipline requirement ensures constraints and rationale never mingle and "truth lives" in the correct artifact. Validation at execution time prevents drift. | 2026-05-03 |
| CLI is workflow enforcer, not code generator or agent | PRD Non-Goals state "Not a code generator, agent, or prompt optimizer — enforces workflow, not content creativity." CLI responsibility is guardrails and boundaries, not creative automation. | 2026-05-03 |
| All artifact reads and writes must be local to configured project root; no remote calls or touching non-RAES files | PRD Data Storage & Privacy section: "Reads and writes only to explicitly configured files in the current project root (never touches non-RAES files)" and "All user and slice history data stored locally; no remote calls." Ensures safety, privacy, and clear file ownership. | 2026-05-03 |
| CLI core commands (excluding --history) must execute in under 200ms on commodity hardware | PRD Scalability & Performance: "CLI should execute core commands (other than --history) in under 200ms on commodity hardware." Ensures responsiveness and practical usability in daily workflow. | 2026-05-03 |
| raes.config.yaml must exist and be valid before any execution; config validation is a mandatory pre-flight check | PRD Functional Requirement --check-config and Safety/Drift Guards ensure artifact paths and types are validated before execution begins. Prevents downstream errors and guides self-healing. | 2026-05-03 |
| History and log output must be machine-readable to support future ingestion and analysis | PRD Extensibility/Auditability requirement: "History/log output and flagging designed to be machine-readable for later ingestion/analysis." Enables audit trail and integration with downstream tooling. | 2026-05-03 |
| Flagging mechanism must halt advancement until ambiguity is resolved and flag is cleared by human | PRD User Experience (Advanced Features): "User flags ambiguity: Tool halts advancement until issue is resolved and flag is cleared, ensuring no action 'papers over' uncertainty." Prevents technical debt from buried ambiguity. | 2026-05-03 |
| CLI outputs must be color-coded with clear icons/symbols; error messages must state what is blocking, why, and what file/data needs fixing | PRD UI/UX Highlights require "Strict, concise error messages: always say what is blocking, why, and what file or data needs to be fixed before proceeding." Supports operator clarity and reduces support burden. | 2026-05-03 |
| Artifact file updates must be atomic; never partial writes | PRD Technical Needs: "Safe, atomic file I/O for updating artifacts; never partial writes." Ensures data integrity and prevents corruption during interruption or failure. | 2026-05-03 |
| Slice execution must walk through either Execution Loop or Review Loop as determined by slice type, enforcing boundaries for each artifact update | PRD Core Experience Step 2: "Tool walks user/agent through either Execution Loop or Review Loop (prompting, recording, and enforcing boundaries)." Each slice type has distinct rules; CLI enforces them strictly. | 2026-05-03 |
| CLI must validate that all artifact paths referenced in raes.config.yaml are present and of correct type before execution | PRD Functional Requirement --check-config (Priority: High) ensures missing or misconfigured artifacts are caught early, preventing silent failures during execution. | 2026-05-03 |
| Core RAES execution flow and artifact rules cannot be bypassed, even via subcommands or extensibility | PRD Extensibility/Auditability: "CLI extensible via subcommands, but core RAES execution flow and artifact rules cannot be bypassed." Protects methodology integrity as tooling evolves. | 2026-05-03 |
| Exit codes: 0 = success/help/version; 1 = usage error (unknown option, extra arg, missing required arg); 2 = reserved for config/runtime errors | system.md contract: "Exit code assignments must be recorded in decisions.md when introduced." These codes are a public contract and must not change without an explicit decision entry. | 2026-05-03 |
| Implementation language and runtime: TypeScript with Node.js, `--experimental-strip-types`, no build step, no external runtime dependencies | Matches the raes-init sibling project exactly; keeps toolchain consistent across the RAES CLI suite and avoids adding a separate compile step. | 2026-05-03 |
| `main(argv, io)` is the testable entry point; `io.out`/`io.err` are injectable sinks; the binary entry point at file bottom calls `main` and exits with the returned code | Decouples command logic from process I/O so tests can assert on output without spawning subprocesses. This pattern must be preserved for all future command handlers. | 2026-05-03 |
| `checkConfig` returns `{ errors: ConfigError[]; config?: RaesConfig }`, not a bare `ConfigError[]` | On success the caller needs the parsed config to display verified paths and to support future commands (e.g. `--status`, `--list-slices`) that must load config without re-validating from scratch. Returning a bare array forced callers to re-parse the config themselves, duplicating I/O and validation logic. Any future function that validates and then acts on config should follow the same pattern: return errors alongside the parsed result so the caller never has to re-parse on success. | 2026-05-03 |
| Every `ConfigError` must include a non-empty `fix` string describing the one action that resolves the error | Bare error messages tell the operator what is wrong but not what to do. The PRD requires "strict, concise error messages: always say what is blocking, why, and what file or data needs to be fixed before proceeding." `fix` is the "what to do" half of that contract. Future slices that introduce new validation errors in `extractConfig`, `validatePaths`, or any new validator must populate `fix`; a `ConfigError` without `fix` is incomplete and must not be merged. | 2026-05-03 |
